# Autonomous Migration Architect (AMA)

**AMA** is a Week 1 ingestion and reporting stack for **legacy-to-cloud migration planning**. It turns **SQL execution logs**, **Git-hosted SQL**, and **communications metadata** into a single, explainable picture: which tables and columns matter, how legacy names line up against target DDL, and what still needs human judgment—delivered both as **CLI output** (JSON, Markdown, Excel) and an **interactive Streamlit dashboard**.

This README explains how the pipeline works under the hood and how to run it in your environment.

---

## 1. The vision

Modernizing a data estate is rarely a pure “lift and shift.” Teams need to know **where value concentrates** (query volume, business importance), **how messy the identifiers are** (Hebrew labels, typos, dialect quirks), and **whether proposed canonical mappings are trustworthy** before they cut over to cloud-native schemas.

AMA automates the **first mile** of that work:

- **Ingest** raw SQL text and related signals (comms, Git paths) with defensive parsing and normalization.
- **Resolve** log column names onto a **target DDL vocabulary** using a layered merge strategy (glossary → exact → lexical + semantic retrieval), with explicit **confidence bands** and **human-in-the-loop (HITL)** review for borderline cases.
- **Report** the same truth to engineers **and** managers: executive KPIs, domain clustering, a business-facing glossary, searchable “ask the data,” and a review queue for approvals—aligned with the **Excel workbook** generated by the CLI.

## 2. The ingestion pipeline (“dirty data” slayer)

### 2.1 Sanitization layer

Before parsing or embedding, all SQL and identifiers pass through **`sanitize_text`** / **`sanitize_sql_text`** (`ama/sanitize.py`):

| Concern | What we do |
|--------|------------|
| **Null bytes & control characters** | `\x00` is removed; most C0 controls are stripped (tabs/newlines retained where safe). |
| **Unicode** | Text is normalized to **NFC** so visually identical strings compare and hash consistently. |
| **Identifiers** | **`normalize_sql_identifier`** applies NFC, strips quotes/brackets, collapses whitespace to underscores, **casefolds ASCII** for stable Latin keys, and **preserves non-ASCII** forms (e.g. Hebrew) without mangling them. |
| **RTL display** | Helpers for RTL-heavy identifiers (e.g. mirroring for LTR consoles) keep logs readable without changing semantic keys. |

This layer is what makes downstream **vector embeddings**, **merge keys**, and **JSON reports** stable even when upstream logs are noisy.

### 2.2 SQL parsing and regex fallback

Column and table usage are extracted with **SQLGlot** (`parse_sql_query` in `ama/sql_pipeline.py`). When the dialect or the statement shape causes a **`ParseError`**, or the parse yields no `SELECT` chunks, AMA does **not** silently drop the row: it falls back to **`_fallback_regex_extract`**, a conservative pattern pass that:

- Looks for **`FROM schema.table`**-style fragments and records coarse **`select:*`**-style hits.
- Optionally picks up unqualified **`FROM identifier`** patterns when they are not parser keywords.

The fallback is intentionally **lossy** compared to a full AST walk, but it **recovers signal** from malformed or non-standard SQL that would otherwise disappear from inventory and importance calculations.

## 3. The alias resolver (the bridge)

The **`AliasResolver`** (`ama/alias_resolver.py`) maps **log column names** onto **canonical DDL column names** supplied by your project (JSON DDL lists). Resolution is **ordered** and **auditable**.

### 3.1 Four layers of merging logic

1. **Glossary (highest trust for bilingual / business terms)**  
   If the normalized log token hits the Hebrew/English (or extended) glossary and the target exists in the DDL list, AMA emits a **glossary** strategy with **high confidence** (~0.95) and a citation that records the business alignment.

2. **Exact DDL match**  
   After NFC + identifier normalization, if the log identifier equals a DDL name, AMA records **exact_ddl** (~0.98 confidence). This is the unambiguous “same name, same column” path.

3. **Lexical similarity**  
   For fuzzy cases, AMA computes **lexical similarity** between the log token and each DDL candidate (e.g. `difflib`-style alignment). This anchors English-like typos and near-misses so unrelated DDL columns do not “win” purely from embedding noise.

4. **Semantic vector retrieval**  
   Hash-based embeddings (`hash_embedding` on sanitized prompts) score each log/DDL pair. For moderate DDL sizes, **pairwise** vector and lexical scores are **blended** (`0.55 × vector + 0.45 × lexical`) to rank candidates. For very large DDL sets, a **vector store** path retrieves top DDL hits instead of full pairwise scans.

An optional **LLM reasoner** hook can adjust confidence when configured; the default offline path uses the deterministic blend above.

**RTL safety:** For right-to-left scripts without glossary or exact coverage, AMA **caps** confidence and applies **sanity checks** so Hebrew tokens are not blindly glued to arbitrary English DDL via vector similarity alone.

### 3.2 Hard threshold: merge floor (default **0.4**)

Not every scored candidate should become a migration fact. AMA uses **`merge_confidence_floor`** (default **0.4**, configurable via **`AMA_MERGE_CONFIDENCE_FLOOR`** or CLI **`--merge-floor`**):

- Below the floor (or when sanity checks fail), mappings fall into **trash** or **review** buckets—not **confirmed** merges.
- Separately, **`merge_confirmed_threshold`** (default **0.8**, **`AMA_MERGE_CONFIRMED_THRESHOLD`**) gates when a **vector-backed** match is allowed to promote into **confirmed** merged entities; glossary and exact matches bypass that bar by design.

Together, these thresholds **protect migration integrity**: weak or ambiguous links surface as **review** (HITL) or **trash**, not as false “done” in the Excel **Migration** sheet.

## 4. Interactive Streamlit dashboard (managerial UI)

Launch the UI with:

```bash
ama-dashboard --report-path path/to/report.json
```

You can also set **`AMA_REPORT_PATH`** or upload a JSON file from the sidebar. The dashboard consumes the **same JSON structure** as the CLI Excel export.

### 4.1 Executive overview

- **KPIs:** `%` confirmed (merge scope), **queries matched**, and **confirmed column** counts.
- **Impact vs. readiness:** A **scatter plot** of **importance** (business value) vs **merge confidence** (readiness), with bubble size tied to **query volume**—highlighting high-value, high-confidence tables to migrate first.
- **Domain importance:** When discovery payload includes an executive **domain matrix**, a **bar chart** summarizes **business importance** and **migration complexity** by domain.

### 4.2 Domains (domain explorer)

**Domain deep dives** group the portfolio by **business domain** (e.g. Finance, CRM). For each domain you see **data health**: confirmed vs review vs trash counts, average importance, and a **risk level**, plus a filterable inventory of tables in that domain.

### 4.3 Business glossary

The **Business Translator** turns raw merge rows into **manager-readable cards**: legacy columns, target DDL, confidence, and **grouped source tables** when the same mapping appears across multiple tables. A summary table and **expandable detail rows** (with confidence gauges) keep both overview and drill-down in one place.

### 4.4 Ask the data

**Heuristic semantic search** over tables, column mappings, and glossary cards—**Hebrew or English**, with synonym expansion and Unicode normalization (e.g. concept queries for money/amount terms).

### 4.5 Tables

Search and inspect **per-table** inventory: domain assignment, **business descriptions** from fact sheets when present, and the **alias merge** breakdown—**confirmed → DDL**, **review**, and **trash / low-signal** rows.

### 4.6 Review (HITL)

The **human-in-the-loop** queue lists **review-band** merge suggestions. For each row you can **Approve** or **Reject**. Decisions persist in a **sidecar file** next to the report: **`<report_stem>.hitl.json`**.

**Important:** The live JSON report is **not** automatically rewritten when you approve in the UI. To push approvals into **`alias_merge.merged_entities`** (and therefore into **Excel Migration** / downstream JSON), run:

```bash
ama-ingest apply-hitl --report path/to/report.json --format excel
```

This writes **`report.with_hitl.xlsx`** (or use `-o` / `--format json` for a merged JSON). Point the dashboard at the merged JSON if you want the executive view to reflect post-HITL state.

## 5. Setup and deployment

### 5.1 Prerequisites

- **Python ≥ 3.11**
- Install the package (from the repository root):

```bash
pip install -e .
```

Optional: for heavier embedding models (not required for the default hash embeddings), install **`[embed]`** extras per `pyproject.toml`.

### 5.2 Environment variables (AMA\_ prefix)

Configuration uses **`pydantic-settings`** with prefix **`AMA_`** (see `ama/config.py`). Common variables:

| Variable | Purpose |
|----------|---------|
| **`AMA_TARGET_SCHEMA`** / **`AMA_TARGET_TABLE`** | Default focus table for single-table ingestion (`schema.table`). |
| **`AMA_DEFAULT_DB`** | Catalog/database name when logs only reference `schema.table` (also inferred from data-root folder name when unset). |
| **`AMA_MERGE_CONFIDENCE_FLOOR`** | Default **0.4** — minimum score to treat a vector path as non-trash; CLI `--merge-floor` overrides per run. |
| **`AMA_MERGE_CONFIRMED_THRESHOLD`** | Default **0.8** — bar for promoting vector matches to confirmed merges. |
| **`AMA_DDL_COLUMNS_PATH`**, **`AMA_GLOSSARY_PATH`**, **`AMA_SQL_LOGS_GLOB`**, **`AMA_COMMS_DIR`**, **`AMA_GIT_SQL_ROOTS`** | Project layout and inputs (defaults point at `sample_data/`). |
| **`AMA_QDRANT_PATH`** | Optional on-disk Qdrant persistence for vector stores. |
| **`AMA_OPENAI_API_KEY`** or **`OPENAI_API_KEY`** | Optional — enriches business labels / narratives in `business_logic.py` when set (`AMA_OPENAI_MODEL` defaults to `gpt-4o-mini`). |
| **`AMA_REPORT_PATH`** | Convenience default path for **`ama-dashboard`** when not passing `--report-path`. |

A **`.env`** file in the working directory is loaded automatically when present.

### 5.3 CLI commands

**Run ingestion** (SQL logs, comms, Git, importance, optional discovery):

```bash
ama-ingest run --help
```

**Export JSON / Markdown / Excel:**

```bash
ama-ingest run --format json -o report.json
ama-ingest run --format excel -o report.xlsx
```

**Apply HITL sidecar to the report** (see §4.6):

```bash
ama-ingest apply-hitl --report report.json --format excel -o report.with_hitl.xlsx
```

**Launch dashboard:**

```bash
ama-dashboard --report-path report.json
```
